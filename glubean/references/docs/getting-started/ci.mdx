# Connect CI

Use the same profile locally and in CI. Define a `ci` profile in `glubean.yaml`
once, then run it with one command everywhere — that is the simplest way to
avoid drift.

## The `ci` profile

`glubean init` scaffolds a `ci` profile. It declares which suites run, plus
fail-fast, reporters, and (optionally) thresholds:

```yaml
profiles:
  ci:
    suites: [tests]
    execution:
      failFast: true
    reporters:
      junit: .glubean/results/junit.xml
      resultJson: .glubean/results/ci.result.json
```

## Recommended command

```bash
npx glubean ci run
```

`glubean ci run` is shorthand for `glubean run --profile ci`. Suites,
fail-fast, and reporter paths all come from `profiles.ci` — nothing to
remember on the command line, and CI matches what you run locally with
`glubean run --profile ci`.

- `suites: [tests]` keeps CI focused on committed regression tests.
- `reporters.resultJson` gives you a stable machine-readable artifact.
- `reporters.junit` integrates with common CI test dashboards.

Override a path ad-hoc when needed: `glubean ci run --reporter junit:other.xml`.

## GitHub Actions example

```yaml
name: test-api
on: [push, pull_request]

jobs:
  glubean:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "22"
      - run: npm ci
      - run: npx glubean ci run
```

## When you are ready for uploads

Add `--upload` to send results to Glubean Cloud after each CI run:

```bash
npx glubean ci run --upload
```

This requires `GLUBEAN_TOKEN`. Setup is covered in [Upload Your First Run](/getting-started/upload-to-cloud).
